# Irrigation Computer – Home Assistant Custom Integration

A multi-zone irrigation controller for Home Assistant. Replaces the previous
single-zone YAML/package solution (`packages/irrigation_calculator/*`) with a
proper integration that exposes one device per zone, supports any number of
zones and removes all hard-coded entity IDs.

Key features:

- **Multi-zone**: as many zones as you like, each with its own relay, power
  sensor, watering duration, growth phase, thresholds and fallback window.
- **Radiation-driven trigger**: a single global solar-radiation sensor is
  trapezoidally integrated to Wh/m². Each zone tracks its own
  *radiation-since-last-irrigation* counter that resets when that zone runs.
- **Soil-moisture trigger** *(optional per zone)*: attach any number of
  moisture sensors; when their average stays below a configurable threshold
  for a configurable dwell time, the zone starts. Independent of the
  radiation trigger – enable either, both or neither.
- **Minimum-interval guard**: per zone, a configurable minimum interval
  between two automatic runs applies to radiation, moisture and fallback
  triggers (manual start always wins).
- **Fallback trigger**: per zone, configurable interval + active time window.
- **Manual control**: per-zone start/stop button + service + writable
  *Run* switch.
- **Power monitoring**: per zone, with explicit `power_entity_id` or automatic
  derivation from the relay name (`sensor.<switch>_power`). Notifies on
  undercurrent / overcurrent / power-after-stop.
- **Robustness**: `asyncio.Lock` per zone, watchdog forcing relay-off on
  failures, graceful behaviour for `unavailable`/`unknown` sensors, defensive
  relay-off after HA restarts mid-run, persistent storage of run state.
- **Native UI**: every config knob is also exposed as a Number/Select/Time/
  Switch entity – no more `input_*` helpers required.
- **HACS-ready**.

## Requirements

- Home Assistant 2024.12.0 or newer (uses the `time` platform and modern
  selector API).
- A `switch.*` entity per zone (smart plug, ESPHome relay, Shelly, ...).
- Optional: a sensor providing solar irradiance in **W/m²** or **Wh/m²**.

## Installation

### Via HACS (recommended)

1. HACS → Integrations → top-right menu → *Custom repositories*.
2. Add `https://github.com/DevelopementSoniverse/irrigation-computer` as
   category *Integration*.
3. Install **Irrigation Computer**, restart Home Assistant.
4. Settings → Devices & Services → *Add Integration* → "Irrigation Computer".

### Manual

Copy `custom_components/irrigation_computer/` into your `config/custom_components/`
directory and restart Home Assistant.

## First-time setup

1. The initial flow only asks for the optional **global radiation source
   sensor** and its unit. Skip it if you don't have one yet.
2. Open the integration's **Configure** dialog (Settings → Devices & Services).
   The options menu lets you:
   - Edit global settings (radiation source and optional smartphone push alerts).
   - **Add a zone**.
   - Edit an existing zone.
   - Delete a zone.

For each zone you provide a name, the relay switch entity, an optional power
sensor, a watering duration in seconds, the growth phase + the three radiation
thresholds (planting / fruit set / ripening), an alert delay for startup power,
under-/overcurrent thresholds and a fallback window.

## Entities per zone

The zone name is used as the device name; entity IDs are generated by HA from
the device name + the entity name below.

| Platform       | Entity                         | Purpose |
|----------------|--------------------------------|---------|
| binary_sensor  | Running                        | Zone is currently watering |
| binary_sensor  | Fallback active                | Now is inside the fallback window |
| binary_sensor  | Relay available                | Underlying switch is reachable |
| binary_sensor  | Relay error                    | Last run reported a relay/power problem |
| sensor         | Time since last irrigation     | minutes |
| sensor         | Radiation since last irrigation | Wh/m² |
| sensor         | Current radiation threshold    | Wh/m² (depends on phase) |
| sensor         | Current power                  | W (proxy to power_entity) |
| sensor         | Runs 24h                       | count |
| sensor         | Last run                       | timestamp |
| sensor         | Last reason                    | Manual / Radiation / Fallback / Soil moisture |
| sensor         | Average soil moisture          | % (mean of configured moisture sensors) |
| switch         | Run                            | Manual on/off |
| switch         | Fallback enabled               | Toggle fallback trigger |
| switch         | Radiation trigger enabled      | Toggle radiation trigger |
| switch         | Soil moisture trigger enabled  | Toggle soil-moisture trigger |
| button         | Start irrigation               |  |
| button         | Stop irrigation                |  |
| button         | Reset radiation counter        |  |
| number         | Watering duration              | s |
| number         | Threshold Planting / Fruit Set / Ripening | Wh/m² |
| number         | Power alert delay              | s |
| number         | Undercurrent / Overcurrent threshold | W |
| number         | Max runs 24h alert threshold   | count (0 disables alert) |
| number         | Fallback interval              | min |
| number         | Soil moisture threshold        | % |
| number         | Soil moisture dwell time       | min (how long the average must stay below the threshold) |
| number         | Minimum interval between runs  | min (applies to all automatic triggers; 0 disables) |
| select         | Growth phase                   | Planting / Fruit Set / Ripening |
| time           | Fallback start / Fallback end  | HH:MM:SS |

There is also one controller-level entity: **Global radiation energy** (Wh/m²,
total integrated reading from the radiation source).

Alerts are always created as Home Assistant persistent notifications. In the
global options, you can additionally enable push alerts and select one or more
Home Assistant mobile app devices that should receive the same alert on their
smartphones.

A complete catalogue of every alert (trigger, effect on the running irrigation
and whether a push is sent) is documented in
[`docs/ALERTS.md`](docs/ALERTS.md) ([deutsche Fassung](docs/ALERTS.de.md)).

## Services

| Service                                 | Fields                       |
|-----------------------------------------|------------------------------|
| `irrigation_computer.start_zone`        | `zone_id`, `reason`          |
| `irrigation_computer.stop_zone`         | `zone_id`                    |
| `irrigation_computer.reset_radiation`   | `zone_id`                    |
| `irrigation_computer.start_all`         | `reason`                     |

`zone_id` is the stable UUID-style ID assigned to each zone when you create it
(visible via the Diagnostics download or in the device's `Identifiers`).

## Auto-managed dashboard

The integration ships an opt-in, self-updating Lovelace dashboard with one
card per zone (plus a controller card).

- The dockerised test setup creates it for you.
- For HACS installs: add a **Storage** dashboard with URL path `irrigation`
  once via *Settings → Dashboards → ➕*. The integration will populate it on
  the next setup/reload and rebuild it whenever you add, edit or delete a
  zone via the options flow.
- Each managed dashboard config carries the marker
  `auto_managed_by_irrigation_computer: true`. Remove it (Raw configuration
  editor) to opt out of auto-rebuilds and keep your manual layout.

## Test environment

A complete dockerised test setup with a pre-provisioned admin user and
simulated relays / radiation sensor / soil moisture sliders lives under
[`docker/`](docker/). The mocks include `sensor.mock_soil_moisture_a` and
`sensor.mock_soil_moisture_b` (each backed by an `input_number` slider
`0–100 %`) so you can verify the moisture trigger end-to-end without any
physical hardware. See [`docker/README.md`](docker/README.md).
